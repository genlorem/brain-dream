## Proposal #1: Предотвратить систематический перерасход cost limit

**Что заметил:** `gemini_cost_usd: 0.508257` при `DREAM_COST_LIMIT_USD=0.50` — перерасход +1.6%. Это не случайность: cost-check явно происходит **после** завершения прохода. При 96 проходах средняя стоимость ≈ $0.0053/pass; последний проход переступил черту, но уже был запущен.

**Предложение:** В `brain-dream.sh`, в цикле запуска Gemini-проходов, перед каждым `launch_gemini_pass` добавить проверку:

```bash
# Оценка: если средний расход на проход + текущий итог превысит лимит — стоп
if (( GEMINI_LAUNCHED > 0 )) && [[ "$DREAM_COST_LIMIT_USD" != "0" ]]; then
  avg_cost_per_pass=$(jq -s '[.[].cost_usd//0] | add/length' "$USAGE_SINK" 2>/dev/null || echo "0")
  projected=$(echo "$gemini_cost_so_far + $avg_cost_per_pass" | bc -l 2>/dev/null || echo "999")
  if (( $(echo "$projected > $DREAM_COST_LIMIT_USD" | bc -l) )); then
    STOP_REASON="cost_limit_projected"
    break
  fi
fi
```

Функция `jq` уже используется повсюду в скрипте; `USAGE_SINK` заполняется дочерними `gemini.sh`. Изменение локальное в одном месте цикла.

**Обоснование:** Устраняет стабильный ~5% перерасход без изменения логики сна. Особенно важно когда cron запускается ежедневно — каждую ночь чуть выходим за $0.50.

**Риск:** низкий. **Обратимость:** легко (убрать три строки).

## Proposal #2: Обрезать кандидатов по confidence перед синтезом

**Что заметил:** `candidate_count: 263` при итоговом ТОП-10. Claude-синтез получает 263 кандидата в один промпт. Большинство имеют дефолтный `confidence: 0.7` (новые из `registry_append`), часть — выше из-за `registry_bump_hit` (+0.05 за повтор). При 263 кандидатах промпт синтеза вероятно > 30K токенов, и Claude выбирает из шума.

**Предложение:** В `brain-dream.sh`, перед построением synthesis-промпта из `CANDIDATES_FILE` — отсортировать по `confidence` desc, взять первые 120:

```bash
# Перед передачей кандидатов в Claude-синтез
CANDIDATES_TOP="$DREAM_OUT_DIR/.candidates-top.jsonl"
jq -s 'sort_by(-.confidence) | .[:120] | .[]' "$CANDIDATES_FILE" \
  > "$CANDIDATES_TOP" 2>/dev/null || cp "$CANDIDATES_FILE" "$CANDIDATES_TOP"
# Далее использовать CANDIDATES_TOP вместо CANDIDATES_FILE в synthesis prompt
```

**Обоснование:** Инсайты, которые встречались в нескольких проходах (`confidence > 0.7`), — это кросс-доменные паттерны. Именно они должны попасть в синтез первыми. При 263 кандидатах с одинаковым confidence=0.7 сортировка не даст результата — это сигнал, что сон «первый раз» видит всё это (все новые), и тогда 120 — уже лучше, чем 263 по качеству синтеза.

**Риск:** низкий. **Обратимость:** легко — заменить `CANDIDATES_TOP` обратно на `CANDIDATES_FILE`.

## Proposal #3: Защита от перегрева маленького пула свежих нод

**Что заметил:** `domains: 6` (`travelmart, personal, marquiz, skvo, indie, govori`), `DREAM_RECENT_WEIGHT_PCT=70`, 96 Gemini проходов. При 70% recency-bias один домен с малым количеством свежих нод (например, `indie` или `govori` обновлялись редко) попадает в 70% каждого прохода своими немногими нодами. Content-hash (`content-hash.sh`) ловит только нормализованные точные совпадения — семантические near-дубли проходят насквозь. Итог: `candidate_count: 263` при реальном разнообразии значительно меньше.

**Предложение:** В `brain-dream.sh`, в `dream-should-run.sh` или в начале сэмплинга — добавить автоматическое снижение `DREAM_RECENT_WEIGHT_PCT` при маленьком глобальном пуле свежих нод:

```bash
# В начале brain-dream.sh, после сбора NODES_FILE
fresh_count=$(awk -F'\t' '$3 >= '"$last_run_epoch"' {c++} END {print c+0}' "$NODES_FILE" 2>/dev/null || echo 0)
total_count=$(wc -l < "$NODES_FILE" 2>/dev/null || echo 1)

if (( fresh_count < 50 )); then
  # Свежих нод мало — снижаем bias, иначе одни и те же ноды в каждом проходе
  effective_recent_weight=$(( DREAM_RECENT_WEIGHT_PCT * fresh_count / 50 ))
  (( effective_recent_weight < 50 )) && effective_recent_weight=50
  log "INFO" "event=recency_bias_reduced fresh=$fresh_count weight=$effective_recent_weight"
  DREAM_RECENT_WEIGHT_PCT=$effective_recent_weight
fi
```

Пороговое значение `50` — примерно половина от `THRESHOLD_NEW_NODES * 2.5`, можно сделать параметром `DREAM_MIN_FRESH_NODES`.

**Обоснование:** Если с прошлого сна в graph добавили 15 нод, то при 70% bias они попадут в каждый из 96 проходов. Эти 15 нод породят ~30–60 кандидатов с разными формулировками но одной семантикой. Понижение до 50% равномерно разбавит выборку историческими нодами и повысит реальное разнообразие инсайтов.

**Риск:** средний — меняет «характер» сна, он станет чуть менее про «сегодняшнее». **Обратимость:** легко, одна переменная.
