## Proposal #1: Алерт о downtime Gemini в Telegram

**Что заметил:** `gemini_passes=0, gemini_cost=0` уже две ночи подряд (Jun 8, Jun 9). Система молча ушла в Sonnet-fallback и каждую ночь сжигает ровно 30% сессионного лимита (60/200 вызовов). В Telegram приходит нормальный дайджест — никакого маркера о том, что Gemini лежит.

**Предложение:** В `brain-dream.sh`, после секции где определяется `gemini_passes_actual=0` и активируется fallback, создавать маркер `$DREAM_OUT_DIR/.gemini-down-since` (если не существует — пишем текущую дату). В Telegram-подпись добавлять `⚠️ Gemini offline Nd` когда `down_days >= 1`.

**Обоснование:** За 5 ночей fallback-подряд — 300 из 200 (сверхлимит невозможен, значит реально 60/ночь × N — это 150+ вызовов дневного окна на сны). Без алерта неясно когда пополнить Gemini API balance.

**Риск:** Низкий. **Обратимость:** Легко.

**Примерный diff** (в `brain-dream.sh`, после подсчёта реальных Gemini-проходов):
```bash
GEMINI_DOWN_MARKER="$DREAM_OUT_DIR/.gemini-down-since"
if (( ${gemini_passes_actual:-0} == 0 )); then
  [[ -f "$GEMINI_DOWN_MARKER" ]] || date -u +%F > "$GEMINI_DOWN_MARKER"
  _down_since=$(cat "$GEMINI_DOWN_MARKER")
  _down_days=$(( ( $(date -u +%s) - $(date -u -d "$_down_since" +%s) ) / 86400 ))
  (( _down_days >= 1 )) && GEMINI_DOWN_ALERT="⚠️ Gemini offline ${_down_days}d — Sonnet fallback active"
else
  rm -f "$GEMINI_DOWN_MARKER"
  GEMINI_DOWN_ALERT=""
fi
# Вставить $GEMINI_DOWN_ALERT в переменную, которая уходит в Telegram-сообщение.
```

## Proposal #2: Early-stop по насыщению кандидатов

**Что заметил:** Jun 3 — 86 passes → 184 candidates ($0.498). Jun 4 — 500 passes → 170 candidates ($0.468). В 6× больше проходов, на 8% меньше кандидатов. Весь уникальный сигнал исчерпывается в первых ~100 проходах, дальше `content_hash_insight` поглощает повторы. Именно это переполнение Gemini spend cap вероятно и привело к нынешнему постоянному fallback.

**Предложение:** В pass-цикле `brain-dream.sh` каждые `DREAM_SATURATION_CHECK_INTERVAL=50` проходов сравнивать `wc -l $CANDIDATES_FILE` с предыдущим чекпоинтом. Если `delta < DREAM_SATURATION_MIN_YIELD` (дефолт 3) — выход из цикла.

**Обоснование:** Jun 4 остановился бы на ~100-120 проходах вместо 500. Экономия ~$0.35 = 70% ночного Gemini-бюджета. Spend cap наступал бы значительно реже.

**Риск:** Средний (порог надо подбирать; слишком низкий — обрезает поздние кандидаты из редких доменов). **Обратимость:** Легко — `DREAM_SATURATION_MIN_YIELD=0` отключает проверку.

**Примерный diff** (в основной pass-loop `brain-dream.sh`):
```bash
# Рядом с объявлением DREAM_MAX_RUNS (~строка 100):
DREAM_SATURATION_MIN_YIELD="${DREAM_SATURATION_MIN_YIELD:-3}"
DREAM_SATURATION_CHECK_INTERVAL="${DREAM_SATURATION_CHECK_INTERVAL:-50}"

# В теле pass-цикла (инициализация перед циклом):
_sat_prev=0; _sat_since=0

# Внутри цикла, после записи кандидатов:
(( _sat_since++ ))
if (( DREAM_SATURATION_MIN_YIELD > 0 && _sat_since >= DREAM_SATURATION_CHECK_INTERVAL )); then
  _cur=$(wc -l < "$CANDIDATES_FILE" 2>/dev/null || echo 0)
  _delta=$(( _cur - _sat_prev ))
  if (( _delta < DREAM_SATURATION_MIN_YIELD )); then
    log_msg INFO "event=early_stop reason=saturation delta=$_delta"
    break
  fi
  _sat_prev=$_cur; _sat_since=0
fi
```

**Обоснование:** −20 вызовов/ночь = −10% дневного сессионного лимита. При текущей ситуации (Gemini down уже 2 дня) это накапливается быстро. `DREAM_SYNTH_TOP_N=120` остаётся не задетым — synthesis берёт top-120 из ~160 вместо ~170, разница в хвосте, не в TOP-10.
