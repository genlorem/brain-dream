## Proposal #1: Неправильный путь для домена `personal` в триггере

**Что заметил:** В метаданных снов — `domains: [travelmart, personal, marquiz, skvo, indie, govori, infra]`. При этом в документации `brain-dream.sh` явно написано: `personal -> $HOME/life/brain`. А в `dream-should-run.sh` для всех доменов используется один шаблон:

```bash
root="$HOME/brain/$domain/nodes"   # строка ~65
```

Для `personal` это даёт `~/brain/personal/nodes` — путь не существует. Счётчик `new_count` для `personal` всегда 0. Триггер «видит» новые ноды только из остальных доменов и срабатывает почти исключительно по `time_threshold`.

**Предложение:** Добавить маппинг в `dream-should-run.sh` перед строкой `root=`:

```bash
# dream-should-run.sh, заменить строку root= в цикле:
domain_path() {
  case "$1" in
    personal) printf '%s' "$HOME/life/brain" ;;
    *) printf '%s' "$HOME/brain/$1/nodes" ;;
  esac
}

for domain in $DREAM_DOMAINS; do
  root="$(domain_path "$domain")"
  [[ -d "$root" ]] || continue
```

Либо вынести маппинг в `lib/domain-paths.sh` и source'ить из обоих скриптов.

**Обоснование:** Из-за бага триггер занижает `new_count` и полагается на `time_threshold_96h`. Это объясняет почему все три сна выходят примерно раз в 2-3 дня — не по знанию, а по таймауту.

**Риск:** низкий. **Обратимость:** легко — однострочная правка.

## Proposal #2: `registry_compact` нигде не вызывается — реестр растёт неограниченно

**Что заметил:** Сон 2026-06-26 собрал 437 кандидатов (против 173-176 в предыдущих). 14-дневное окно × 3 сна/нед × ~300 кандидатов = ~1200+ записей в `.insight-hashes.jsonl`. При этом каждый вызов `registry_has_hash` делает полный scan через `jq -e select(...)` — отдельный jq-процесс на каждый из 437 кандидатов. Функция `registry_compact` написана в `lib/insight-hashes.sh:80` но не вызывается нигде в коде.

**Предложение:** Добавить вызов в конец прогона `brain-dream.sh` (после записи финального markdown, до Telegram):

```bash
# brain-dream.sh, в финальной секции после stage=synthesize:
log_msg "INFO" "compacting insight registry"
registry_compact
```

Дополнительно — для ускорения lookup в рамках одной ночи: один раз в начале прогона загружать хэши в flat-файл и делать `grep -qF "$h" "$HASH_FLAT"` вместо 437 jq-запусков:

```bash
# В начале прогона, однократно:
HASH_FLAT="$(mktemp)"
jq -r '.hash' "$INSIGHT_REGISTRY" 2>/dev/null > "$HASH_FLAT" || true
```

**Обоснование:** При 437 кандидатах jq поднимается 437 раз с полным чтением файла. Это накапливается. После compact + flat-lookup ночь ускорится, реестр не пухнет.

**Риск:** низкий. **Обратимость:** легко.

## Proposal #3: Pre-synthesis cap при взрывном росте кандидатов

**Что заметил:** Скачок 176 → 437 кандидатов произошёл при переходе на CLI-backend (200 проходов, cost=0). В синтез идут все кандидаты разом — именно это уже ломало прогон (`0fda83d: синтез падал на длинном промпте`). Топ-10 выбирается из 437 против обычных 175 — качество выбора не растёт линейно с размером пула, зато промпт растёт в 2.5x.

**Предложение:** В `brain-dream.sh`, перед вызовом синтеза (секция где собирается финальный список кандидатов для Gemini/Claude топ-10), добавить lens-aware сэмплинг:

```bash
# brain-dream.sh, перед передачей CANDIDATES_FILE в синтезатор:
DREAM_SYNTHESIS_CAP="${DREAM_SYNTHESIS_CAP:-250}"

if (( $(wc -l < "$CANDIDATES_FILE") > DREAM_SYNTHESIS_CAP )); then
  log_msg "INFO" "capping candidates $(wc -l < "$CANDIDATES_FILE") → $DREAM_SYNTHESIS_CAP (lens-balanced)"
  # равномерно по линзам: sort -R, затем head по капу
  CANDIDATES_CAPPED="$(mktemp)"
  jq -r '.lens' "$CANDIDATES_FILE" | sort -u | while read -r lens; do
    grep -F "\"lens\":\"$lens\"" "$CANDIDATES_FILE"
  done | shuf | head -n "$DREAM_SYNTHESIS_CAP" > "$CANDIDATES_CAPPED"
  CANDIDATES_FILE="$CANDIDATES_CAPPED"
fi
```

(Точный код зависит от формата `CANDIDATES_FILE` в `brain-dream.sh` — переменная скорее всего путь к JSONL с полем `lens`.)

**Обоснование:** 250 кандидатов → топ-10 это уже 25x compression, дальше прирост качества убывает. Lens-balanced сэмплинг сохраняет разнообразие. Синтез-промпт остаётся в безопасных размерах при любом backend.

**Риск:** средний (меняет набор кандидатов в синтезе, топ-10 может чуть измениться). **Обратимость:** легко — убрать блок или выставить `DREAM_SYNTHESIS_CAP=9999`.
