## Proposal #1: Нет алертинга на 6-дневный отказ Gemini

**Что заметил:** Во всех 6 снах недели: `gemini_passes: 0`, `gemini_cost_usd: 0`. Sonnet-фолбэк молча перехватил всю нагрузку (`DREAM_SONNET_FALLBACK=1`). Telegram-дайджест уходит каждую ночь без единого упоминания, что primary LLM мёртв уже неделю.

**Предложение:** В `brain-dream.sh`, в секции сборки TG-сообщения, добавить проверку после Gemini-фазы: если `GEMINI_ACTUAL_RUNS == 0`, вставить строку-предупреждение в начало дайджеста. Для подсчёта consecutive nights — читать `.last-run.json` (там уже есть `trigger_reason`), либо добавить поле `gemini_zero_streak` в state-файл.

```bash
# brain-dream.sh, после завершения Gemini-фазы:
if (( GEMINI_ACTUAL_RUNS == 0 )); then
  TG_PREFIX="⚠️ Gemini недоступен (${GEMINI_ZERO_STREAK:-?} ночей подряд). Работаю на Sonnet-фолбэке.\n\n"
fi
```

**Обоснование:** Silent degradation — худший режим для production cron. Без алертинга можно неделями не замечать, что ключ протух или исчерпан monthly cap.

**Риск:** низкий. **Обратимость:** легко.

## Proposal #2: DREAM\_SATURATION\_CHECK\_INTERVAL несоразмерен лимиту fallback-режима

**Что заметил:** Каждую ночь ровно 60 Sonnet-проходов (жёсткий cap = 30% от 200). `candidate_count` ~170. Математика: 170 кандидатов за 60 проходов ≈ 2.8/проход в среднем. Первая проверка насыщения срабатывает на проходе 50 (interval=50). К тому моменту накоплено ≈142 кандидата, прирост за последние 50 проходов ≈ 142 — несравнимо выше порога `DREAM_SATURATION_MIN_YIELD=3`. Early-stop **никогда** не срабатывает. Система всегда выжигает весь лимит.

**Предложение:** Масштабировать интервал относительно max\_runs, а не абсолютно. В `brain-dream.sh` при инициализации Sonnet-фазы:

```bash
# вместо фиксированного DREAM_SATURATION_CHECK_INTERVAL
EFFECTIVE_SAT_INTERVAL=$(( SONNET_MAX_RUNS / 3 ))
(( EFFECTIVE_SAT_INTERVAL < 10 )) && EFFECTIVE_SAT_INTERVAL=10
```

Для fallback\_runs=60: проверки на проходах 20, 40, 60 — три шанса вместо одного. Также поднять `DREAM_SATURATION_MIN_YIELD` до ~15 (сейчас 3 — порог настолько низкий, что никогда не достигается при живом контенте).

**Обоснование:** Текущие параметры насыщения были, вероятно, откалиброваны под Gemini-режим (500 проходов). В единственном реально работающем сейчас режиме (Sonnet fallback, 60 проходов) они бесполезны.

**Риск:** низкий. **Обратимость:** легко (env-переменные).

## Proposal #3: `dream-should-run.sh` считает новые ноды только по 2 доменам из 7

**Что заметил:** В `dream-should-run.sh`:
```bash
DREAM_DOMAINS="${DREAM_DOMAINS:-travelmart personal}"
```
Но в снах поле `domains: [travelmart, personal, marquiz, skvo, indie, govori, infra]` — 7 доменов. Если cron запускает `dream-should-run.sh` без явного `DREAM_DOMAINS`, счётчик `new_count` учитывает только 2 из 7 доменов. Пять доменов (marquiz, govori, infra и др.) полностью выпадают из trigger-логики.

**Предложение:** Синхронизировать дефолт с `brain-dream.sh`, либо вынести общий default в отдельный source-файл:

```bash
# dream-should-run.sh — изменить строку:
DREAM_DOMAINS="${DREAM_DOMAINS:-travelmart personal marquiz skvo indie govori infra}"
```

Или — если домены конфигурируются централизованно — добавить `source "$ORCHESTRATOR_DIR/dream-defaults.env"` в оба скрипта.

**Обоснование:** Сон сейчас запускается каждую ночь по time\_threshold (≥20ч), поэтому баг скрыт. Но если паттерн нагрузки изменится (мало новых нод в travelmart/personal, много в govori) — `new_nodes`-ветка trigger'а не сработает когда должна.

**Риск:** низкий. **Обратимость:** легко.
