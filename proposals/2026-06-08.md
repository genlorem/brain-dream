## Proposal #1: Ранняя остановка при исчерпании yield (dedup-насыщение)

**Что заметил:** 06-04 — 500 gemini_passes → 170 кандидатов. 06-02 — 86 passes → 185 кандидатов. Больше в 6 раз проходов, *меньше* уникальных инсайтов. К 500-му проходу 14-дневный registry (~1400+ хешей за неделю) блокирует почти всё новое. Gemini-бюджет сгорает на проходах, которые 90%+ дедупятся.

**Предложение:** В `brain-dream.sh` — добавить yield-монитор в главный run-цикл. Каждые `DREAM_YIELD_WINDOW=30` проходов сравнивать `candidate_count` до и после. Если прирост < `DREAM_YIELD_MIN_NEW=3` — выставить `STOP_REASON=yield_exhausted` и выходить на синтез. Оба параметра через env, дефолты консервативные.

**Обоснование:** На 06-04 система потратила ~414 «лишних» проходов (500 vs необходимые ~86). Это ~$0.35 впустую. Ранняя остановка при yield < 10% сохранила бы бюджет и позволила бы пустить его на Sonnet-фазу или увеличить `DREAM_CONCURRENCY`.

**Риск:** низкий. **Обратимость:** легко — `DREAM_YIELD_MONITOR_ENABLED=0` отключает.

**Примерный diff** (в главном loop `brain-dream.sh`, рядом с блоком проверки `RUNS`):
```bash
# Добавить в env-секцию вверху:
DREAM_YIELD_WINDOW="${DREAM_YIELD_WINDOW:-30}"
DREAM_YIELD_MIN_NEW="${DREAM_YIELD_MIN_NEW:-3}"
DREAM_YIELD_MONITOR_ENABLED="${DREAM_YIELD_MONITOR_ENABLED:-1}"

# В run-loop, после инкремента RUNS:
if [[ "$DREAM_YIELD_MONITOR_ENABLED" == "1" ]] && \
   (( RUNS % DREAM_YIELD_WINDOW == 0 && RUNS >= DREAM_YIELD_WINDOW )); then
  prev_count_marker="$DREAM_OUT_DIR/.yield-prev-${RUNS}"
  cur_count=$(wc -l < "$CANDIDATES_FILE" 2>/dev/null || printf '0')
  prev_count=$(cat "$prev_count_marker" 2>/dev/null || printf '0')
  printf '%s\n' "$cur_count" > "$prev_count_marker"
  if (( cur_count - prev_count < DREAM_YIELD_MIN_NEW )); then
    STOP_REASON="yield_exhausted"
    break
  fi
fi
```

## Proposal #2: Per-domain cap в synthesis top-N

**Что заметил:** Три дня подряд (02, 03, 04) — заголовок сна крутится вокруг одной темы: "сложность фронтенд-сборки ЗнИ / дублирование данных". Все 7 доменов в `domains:`, но travelmart-инсайты монополизируют топ-10. Причина: `DREAM_RECENT_WEIGHT_PCT=70` + активный спринт → travelmart-узлы свежее → выше confidence → вытесняют personal/marquiz/govori в синтезе.

**Предложение:** В `brain-dream.sh`, в блоке формирования synthesis-промпта (где берётся `DREAM_SYNTH_TOP_N=120` кандидатов) — добавить per-domain ограничение через `DREAM_SYNTH_DOMAIN_CAP`. Логика: если домен претендует на >CAP мест в топ-N, урезать до CAP и добрать из следующих доменов. Дефолт: `DREAM_SYNTH_DOMAIN_CAP=30` (25% от 120).

**Обоснование:** Разнообразие доменов — одна из ключевых ценностей «сна». Когда один домен занимает 60+ из 120 слотов, синтез Claude видит однобокую картину и выдаёт однотемный инсайт. CAP не убирает travelmart-доминирование полностью (30 из 120 — всё ещё много), но даёт шанс personal/indie/govori.

**Риск:** средний — если в travelmart реально важное (крупный инцидент), CAP его урежет. Поэтому дефолт 30, не 15. **Обратимость:** легко — `DREAM_SYNTH_DOMAIN_CAP=0` отключает.

**Примерный diff:**
```bash
# Добавить в env-секцию:
DREAM_SYNTH_DOMAIN_CAP="${DREAM_SYNTH_DOMAIN_CAP:-30}"

# В synthesis-подготовке, вместо простого sort+head:
# Было:
top_candidates=$(sort -rn -t'"' ... | head -n "$DREAM_SYNTH_TOP_N")
# Стало (через jq с per-domain группировкой):
top_candidates=$(jq -s --argjson cap "$DREAM_SYNTH_DOMAIN_CAP" --argjson n "$DREAM_SYNTH_TOP_N" '
  sort_by(-.confidence)
  | group_by(.domain)
  | map(if $cap > 0 then .[:$cap] else . end)
  | flatten
  | sort_by(-.confidence)
  | .[:$n]
' "$CANDIDATES_FILE")
```

## Proposal #3: Масштабировать NREM_PASSES пропорционально target_runs в Sonnet-фолбэке

**Что заметил:** 06-08 — Gemini упал (spend cap), Sonnet-фолбэк сделал 60 проходов. `DREAM_NREM_PASSES=20` (первые 20 = узкие линзы: problem/gap/stalled). В нормальном режиме 20/500 = 4% NREM. В фолбэке: 20/60 = **33% NREM**. Результат — сон «Задача VIP-7575 непонятна исполнителю» вместо кросс-доменного синтеза. Narrow lenses передавили.

**Предложение:** В `brain-dream.sh`, при запуске Sonnet-фолбэка — переопределять `effective_nrem_passes` как `max(5, floor(total_target * 0.10))`. При 60 проходах: 6 NREM вместо 20. При 500: 50 (текущее поведение почти не меняется). Управляется через `DREAM_NREM_PASSES_RATIO=0.10`.

**Обоснование:** NREM — «консолидирующая» фаза, REM — «творческая». При малом окне (60 Sonnet-вызовов, 30% сессии) критично получить разнообразие, а не концентрацию на задачах. Фиксированное 20 было нормально при 86-500 Gemini проходах, но ломает баланс в fallback-режиме.

**Риск:** низкий. Только меняет threshold перехода NREM→REM внутри фолбэка, не сам набор линз. **Обратимость:** легко — `DREAM_NREM_PASSES_RATIO=0` → всегда фиксированный `DREAM_NREM_PASSES`.

**Примерный diff** (в `brain-dream.sh`, в блоке инициализации Sonnet-фолбэка):
```bash
# Добавить в env-секцию:
DREAM_NREM_PASSES_RATIO="${DREAM_NREM_PASSES_RATIO:-0.10}"

# В Sonnet-fallback init (рядом с DREAM_SONNET_FALLBACK_RUNS):
if [[ "$RUN_ENGINE" == "sonnet" && -n "$DREAM_NREM_PASSES_RATIO" ]]; then
  computed_nrem=$(echo "$DREAM_SONNET_FALLBACK_RUNS $DREAM_NREM_PASSES_RATIO" | \
    awk '{n=int($1*$2); print (n<5)?5:n}')
  DREAM_NREM_PASSES="$computed_nrem"
  log_msg "INFO" "nrem_passes adjusted to $DREAM_NREM_PASSES for sonnet fallback"
fi
```
